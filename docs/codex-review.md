## 1. 누락 / 모호한 항목 (Critical Gaps)

### 1.1 큐피드 적중 로직 미정의
현재 문서에는 "큐피드 발사"와 "큐피드 적중 알림"만 있고, 적중 조건이 상호 선택인지, 관리자 개입인지, 호감도 누적치 기반인지 없다. 이 상태로는 백엔드 이벤트 모델과 알림 UX를 설계할 수 없다.

### 1.2 호감도 익명/실명 정책 부재
호감도가 익명인지 실명성인지, 수신자가 누구에게 받았는지 알 수 있는지 빠져 있다. 익명이라면 랭킹 공개가 조롱과 추측 놀이로 변질될 수 있고, 비익명이라면 사실상 공개 고백 시스템이 되어 파티 분위기를 망칠 수 있다.

### 1.3 1차 / 2차 신청 의미 불명
1차 신청 / 2차 신청의 의미가 문서상 전혀 설명되지 않는다. 사용자에게 보여줄 이유, 전환 시점, 누가 변경하는지, 이 상태가 어떤 기능 제한과 연결되는지 없어서 UI 문구와 운영 로직을 만들 수 없다.

### 1.4 상대 식별 방식 부재
닉네임 기반 1:1 전송이라고 해놓고 상대 식별 방식이 없다. 현장 참가자 목록을 전부 노출하는지, 검색형인지, 테이블 번호/배지 번호와 매핑하는지 없으면 실제로는 "누구에게 보낼 수 있는지" 자체가 불명확하다.

### 1.5 실시간 알림 전달 방식 미정
모바일 웹에서 브라우저가 닫혀 있거나 백그라운드일 때 푸시를 못 받는데, 소리/배지/토스트 중 무엇을 지원하는지 없으면 "실시간"이 이름뿐인 기능이 된다.

### 1.6 중복 참가 / 재입장 처리 미정
QR 진입 후 같은 사람이 여러 기기에서 들어오면 큐피드/호감도 횟수를 복제할 수 있고, 세션 종료 후 재입장 시 기록을 이어받는지 초기화하는지도 명시가 없다.

## 2. 리스크 (실제로 터질 가능성 높은 것)

### 2.1 일정 압박
2026-04-20 작성, 2026-04-30 납기 기준이면 사실상 개발·기획·테스트를 모두 포함해 10일 남짓이다. 그런데 Day 3에 중간 데모, Day 7~9에 관리자 구현까지 잡아놔서 명세 확정이 하루만 밀려도 통합 테스트가 증발할 가능성이 높다.

### 2.2 현장 네트워크 실패
모바일 웹 + WebSocket 조합은 파티 현장 네트워크에서 가장 먼저 깨진다. 지하/실내 행사장에서 와이파이 품질이 낮으면 실시간 카운터, 알림, 랭킹 갱신이 지연되고 사용자는 "버그인지 아닌지" 구분 못 한 채 이탈한다.

### 2.3 전체 쪽지 열람의 법적 리스크
운영자 입장에선 편하지만, 참가자 입장에선 익명 메시지 검열 서비스로 보일 수 있다. 개인정보처리방침·이용고지 없이 운영하면 "익명인 줄 알고 보냈는데 관리자가 다 본다"는 반발이 실제로 터질 수 있다.

### 2.4 닉네임 사칭
닉네임만으로 운영하면 현장에서 사칭이 매우 쉽다. 인기 있는 사람 닉네임을 선점하거나 유사 닉네임을 만들면 메시지 오발송, 호감도 왜곡, 분쟁이 생기는데 현재 설계엔 본인 식별 장치가 없다.

### 2.5 호감도 순위의 공개 서열화 부작용
MVP에서 재미 요소 같지만, 오프라인 파티에서는 공개 서열화로 작동한다. 특히 하위권 체감, 특정인 몰표, 성비 불균형이 겹치면 참여자 경험이 급격히 나빠지고 운영자가 현장 클레임을 직접 받게 된다.

### 2.6 SOS 운영 책임
SOS 요청은 넣는 순간 운영 책임이 생긴다. 누가 받고 몇 분 내 대응해야 하는지, 장난 호출 필터가 있는지 없으면 실제 안전 기능이 아니라 위험한 가짜 안전장치가 된다.

## 3. MVP 범위 절감 권고

### 3.1 호감도 순위 실시간 공개 → 잘라낼 것
실시간 집계, 동기화, 노출 정책, 부작용 대응까지 비용이 큰데 시장 검증 관점에선 "상호작용이 발생하는가"만 보면 되므로 내부 관리자 대시보드 통계로 대체하는 편이 낫다.

### 3.2 SOS 요청 → 다음 단계로 연기
노래 요청과 SOS는 성격이 완전히 다르고 QA 포인트도 다르다. MVP에선 노래 요청만 남기고 SOS는 운영 프로세스 정리 후 다음 단계로 미루는 게 맞다.

### 3.3 관리자 전체 쪽지 조회 → 신고건만
MVP에서 필요한 건 전체 열람이 아니라 신고 접수된 메시지 확인과 금칙어/차단 정도다. 지금 범위는 법적·운영 리스크만 키운다.

### 3.4 닉네임 변경 로그 조회 → 후순위
변경 3회 제한만 있어도 운영은 가능하고, 로그 조회 UI를 만드는 시간 대비 검증 가치가 낮다.

### 3.5 실시간 파티 인원수 표시 → 단순화
실제 접속 세션과 현장 실제 인원이 일치하지 않을 가능성이 높아서 오히려 신뢰를 깎으니, 관리자 내부 참고 수치로만 두는 편이 안전하다.

## 4. 추가로 필요한 페이지 / 기능

### 4.1 입장 전 안내 / 동의 화면
닉네임 등록 전에 익명성 범위, 관리자 열람 가능 여부, 메시지 정책, 신고 가능 여부를 고지하지 않으면 이후 분쟁을 설명할 근거가 없다.

### 4.2 참여자 목록 / 선택 화면
지금 문서만 보면 누구에게 호감/쪽지를 보내는지 진입점이 없어서 핵심 기능이 구현돼도 사용자는 상대를 찾지 못한다.

### 4.3 빈 상태 화면
받은 쪽지 없음, 남은 큐피드 0회, 아직 호감 받은 적 없음 같은 상태를 설계하지 않으면 첫 사용 순간 화면이 썰렁하고, 사용자는 "뭘 해야 하는지" 이해하지 못한다.

### 4.4 신고 / 차단 기능
익명 기반 메시지 시스템에서 이게 없으면 욕설 1건만 터져도 운영자는 전체 쪽지 조회 외에 대응 수단이 없다.

### 4.5 파티 종료 화면
지금은 "자동 또는 수동 나가기"만 있는데, 종료 후 다시 접속하면 무엇이 보이는지, 기록이 유지되는지, 더 이상 메시지를 못 보내는지 명확한 마감 UX가 없다.

### 4.6 관리자 통합 알림 큐
노래 요청, 신고, SOS, 닉네임 문제를 같은 테이블에서 우선순위로 처리할 수 있어야 실제 운영이 되는데, 현재 명세는 기능이 흩어져 있어 현장 대응 속도가 떨어진다.
