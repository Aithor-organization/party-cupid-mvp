## 2. 기획 배경 및 목적

오프라인 파티에서 자연스러운 호감 표현이 어렵다는 문제. 주최자가 **단계를 자유 정의**하고 참여자가 **단계마다 익명으로 호감을 표시**하여 마지막 호감도 단계 기준으로 매칭.

### 검증 가설
1. 주최자는 단계 자유 설계에 가치를 느끼는가?
2. 참여자는 단계별 익명 호감 표시에 반응하는가?
3. 누적 데이터가 본 개발 단계 확장 가치를 보여주는가?

### 성공 지표
- 닉네임 등록률 ≥ 70% / 호감도 단계당 투표율 ≥ 60%
- 매칭 성사율 ≥ 15% / 단계 커스터마이징 사용률 ≥ 40%

## 3. 서비스 구조 (3-tier)

```
[주최자 PC/모바일] → 회원가입/방 생성/단계 빌더/QR 발급/라이브 운영
       ↓ QR
[참여자 모바일] → 약관/닉네임/단계 진행/매칭 결과
```

## 4. 핵심 룰 (v4 핵심 변경)

### 4.1 단계 시스템 (v3 유지)

방(Room) → Stage 배열 (순서). 각 Stage:
- 이름 (자유 텍스트)
- 호감도 투표 ON/OFF (체크박스)
- 진행 종료 트리거 3종 자유 조합 (⏱ 타이머 / ✅ 전원완료 / 👤 수동)
- 설명 (선택)
- **다중선택 한도 (단계별 설정, v4 신규)** — 호감도 단계만

### 4.2 진행 트리거 충돌 규칙 (v4 명문화)

```
1. 단계 진행 중 트리거 발동 → 첫 번째 발동된 것만 처리 (서버 lock)
2. 주최자 [다음 단계로] 수동 클릭 → 5초 카운트 모달:
   ┌─────────────────────────────────┐
   │ 5초 후 강제 마감됩니다           │
   │ N명 투표 중 (현재 18/23 완료)    │
   │ 5...4...3...2...1                │
   │ [지금 마감] [취소]               │
   └─────────────────────────────────┘
   - 5초 안에 [지금 마감] → 즉시 종료
   - 5초 후 자동 종료
   - [취소] 누르면 트리거 무효화
3. 한 번 단계 종료 → 모든 트리거 비활성화
4. 종료 시각 직전 도착 투표 = 인정, 직후 = 거절
```

### 4.3 매칭 판정 (v3 유지)

마지막 호감도 단계의 양방향 하트 매칭. 이전 호감도 단계 데이터는 보존하되 본인만 참고용으로 봄.

### 4.4 데이터 익명성 (v4 명확화)

```
🔒 다른 참여자에게: 영구 익명 (매칭 시 시스템 자동 공개만 예외)

🔓 주최자에게: 모든 데이터 열람 가능 (운영 권한)
   - 누가 누구에게 하트 줬는지 전부 확인
   - 모든 쪽지 내용 열람
   - 매칭 결과 + 계산 과정
   - CSV 익스포트 (위 모든 데이터)

❌ 주최자도 금지:
   - 표 직접 수정 (API 자체 미제공)
   - 표 삭제/추가/대리 투표
   - "투표 조작" 일체 불가
```

**참여자 약관에 명시 필수**: "주최자는 모든 데이터를 열람할 수 있습니다 (조작은 불가)" — 법적 보호

### 4.5 표 잠금 정책 (v3 유지)

호감도 단계 종료 = 표 영구 잠금. 참여자 + 주최자 모두 수정 불가.

### 4.6 단계 구성 변경 (v4 강화)

```
[주최자] [⚙ 구성 편집] 클릭
   ↓
모든 참여자 → 잠금 화면 ("🛠 현재 이벤트 구성 중입니다")
   ↓
[주최자] 편집 가능 범위:
   - ✅ 종료된 단계: 표시만, 편집 불가 (잠금)
   - ✅ 현재 진행 단계: 삭제 가능 (v4 추가) / 수정 가능
   - ✅ 미래 단계: 자유 추가/삭제/순서 변경
   ↓
[주최자] [구성 완료] → 시작 단계 선택 모달:
   ◯ 종료된 단계 (재오픈 OR 읽기전용 모드 선택)
   ◉ 현재/미래 단계 (자유 선택)
   ↓
모든 참여자 일제히 선택된 단계로 이동
```

### 4.6.1 진행 중 단계 삭제 후처리 (v5 신규)

주최자가 진행 중인 호감도 단계를 삭제할 때:

```
[주최자] 진행 중 단계 [🗑] 클릭
   ↓
2단계 확인 모달:
┌──────────────────────────────────┐
│ ⚠ 진행 중 단계 삭제                │
│ "단계 5: 두번째 투표"               │
│                                  │
│ 영향:                              │
│  • 표 12건 영구 잠금 (DB 보존)      │
│  • 통계에서 제외 (CSV에는 포함)     │
│  • 매칭 계산에서 제외               │
│  • 감사 로그에 삭제 시각 기록       │
│                                  │
│ 확인 문구 입력: ⬚ 삭제 _______      │
│                                  │
│ [취소]            [삭제]           │
└──────────────────────────────────┘

처리 정책:
1. 표 데이터: DB에 영구 보존 (Vote.is_locked=true 유지)
2. 통계 표시: 사용자/주최자 UI에서는 숨김 (단계 자체가 삭제됨)
3. 매칭 계산: 이 단계 제외 (마지막 호감도 단계가 다른 단계로 이동)
4. CSV 익스포트: 삭제된 단계 표 + "deleted_at" 컬럼 포함
5. 감사 로그: 누가 언제 삭제했는지 영구 기록
```

### 4.7 [이전 단계로] 모드 선택 (v4 신규)

주최자가 [이전 단계로] 클릭 시:

```
┌────────────────────────────────────┐
│ 이전 단계로 어떻게 이동할까요?       │
│                                    │
│ ◉ 재오픈 (참여자가 다시 투표 가능)  │
│   ⚠ 새 표가 추가되며 잠금 해제      │
│                                    │
│ ○ 읽기전용 (확인만, 표 변경 X)      │
│   ✓ 기존 표 유지, 안전              │
│                                    │
│ [취소]              [이동]          │
└────────────────────────────────────┘
```

→ DB: `Stage.status = 'reopened' | 'view-only' | 'closed'`

### 4.8 "전원 완료" 정의 (v4 명확화)

```
모수 (분모):
  단계 시작 시점에 입장 중인 참여자 수

분자:
  해당 단계에서 한 번이라도 액션한 사람 수
  (하트 1개 이상 OR "투표 안 함" 명시 선택)

신규 입장 정책:
  - 호감도 단계 진행 중: ❌ 입장 차단 ("다음 단계 시작 시 입장 가능")
  - 비호감도 단계: ✅ 자유 입장

강퇴/이탈자 처리:
  - 기록 영구 보존 (이미 한 행위는 DB에 남음)
  - 즉시 모수에서 제외 (예: 23명 → 22명, 분모 갱신)
  - 분자도 그 사람이 했던 액션 -1
  - 다음 단계부터 불참자 처리

예시:
  단계 5 시작 시 23명 중 → 18명 액션 → 18/23
  진행 중 1명 강퇴 → 17/22 (또는 18/22 if 강퇴자가 액션 안 했었음)
  남은 22명 모두 액션 → ✅ 전원 완료 → 자동 종료
```

### 4.9 다중 선택 한도 (v4 신규)

```
방 생성 시 또는 호감도 단계 편집 시:
  "이 단계에서 최대 몇 명까지 선택할 수 있나요?"
  ⬚ 기본값 3 (1~10 범위)

설정 범위: 호감도 단계마다 다르게 설정 가능 (단계별)

예시:
  단계 2 (첫인상): 최대 5명
  단계 4 (두번째): 최대 3명
  단계 6 (마지막): 최대 1명 (압축)
```

UI: 참여자가 한도 도달 시 ❤ 버튼 disabled + 안내
```
"이 단계 최대 3명 선택. 변경하려면 다른 사람 ❤ 취소 후 시도"
```

### 4.10 자기 자신 선택 ❌

시스템적으로 차단:
- API: 자기 자신 ID로 하트 → 400 에러
- UI: 자기 프로필에는 ❤ 버튼 자체 없음

### 4.11 차단 기능 ❌ 제거 (v4)

v3까지 있던 사용자 차단 기능을 **완전 제거**.
- 신고 기능만 유지 → 주최자가 강퇴로 처리

### 4.12 보안 / 식별 / 알림 (v3 유지)

- 보안: 유효시간 토큰 + 주최자 승인 옵션
- 식별: 닉네임 + 입장순 번호
- 알림: 앱 내 배지 + 토스트 (푸시 보조)

### 4.13 인증 모델 (v6 신규) — 비대칭 인증

> **핵심**: 주최자는 정식 계정, 참여자는 익명 세션 + 닉네임. 두 역할의 사용 패턴이 본질적으로 다르므로 인증 방식도 분리.

| 역할 | 인증 방식 | 식별자 | 세션 수명 |
|------|----------|--------|-----------|
| **주최자** | Supabase Auth (이메일+비밀번호) + OAuth(Google/카카오 옵션) | `auth.users.id` (UUID) | 영구 (refresh token 갱신) |
| **참여자** | Supabase **Anonymous Auth** + 닉네임 입력 | `auth.users.id` (anonymous JWT의 sub) + `nickname` | 7일 (refresh token), 같은 브라우저 한정 |

**왜 비대칭인가**:
- 주최자: 방 재사용 / 데이터 열람 / 추후 결제 → 영속 계정 필수
- 참여자: 한 파티 한 번 쓰고 끝 → 회원가입 마찰이 전환율에 치명적

**왜 Anonymous Auth (브라우저 UUID 아님)**:
- JWT가 발급되어야 RLS(Row Level Security)로 데이터 권한을 강제 가능
- 단순 브라우저 UUID 방식은 누구나 남의 vote를 위조 가능 (보안 붕괴)
- Supabase SDK가 refresh_token을 localStorage에 저장 → 새로고침해도 같은 사용자

**참여자 입장 플로우**:
```
QR 스캔 → /r/:code
  ↓
약관 동의 (주최자 데이터 열람 명시)
  ↓
닉네임 입력 (2~12자)
  ↓
supabase.auth.signInAnonymously()  ← anonymous JWT 발급
  ↓
INSERT participants (
  anon_user_id = jwt.sub,
  nickname = 입력값,
  room_id = :code → id
)
  ↓
파티 홈 진입
```

**세션 복구 / 한계**:
- ✅ 같은 브라우저 새로고침 / 앱 백그라운드 복귀 → 동일 participant로 재연결
- ❌ 다른 브라우저 / 시크릿 모드 닫음 / 다른 디바이스 → 다른 사람으로 인식 (재입장 시 새 닉네임 필요)
- 약관에 "한 기기에서 계속 사용해 주세요" 안내 명시

### 4.14 RLS (Row Level Security) 정책 (v6 신규)

Supabase Postgres의 RLS로 데이터 권한 강제. 백엔드 코드 없이 **DB 레벨에서 차단**.

| 테이블 | 정책 | 설명 |
|--------|------|------|
| `rooms` | host_id = auth.uid() | 주최자만 자기 방 수정 |
| `rooms` | SELECT public (방 코드 알면) | 참여자 입장용 read 허용 |
| `participants` | INSERT: anon_user_id = auth.uid() | 자기 자신만 입장 등록 |
| `participants` | SELECT (참여자 시점): 같은 방의 다른 참여자 닉네임만 read | 실명/이메일 등 추가 정보는 RLS로 차단 |
| `participants` | SELECT (주최자 시점): host_id 매칭 시 전체 read | 주최자 데이터 열람 권한 |
| `votes` | INSERT: voter_id = auth.uid() AND voter_id <> target_id | 자기 자신 ❤ 차단 (4.10 시스템 강제) |
| `votes` | UPDATE/DELETE: ❌ 정책 미생성 = 기본 deny | 표 영구 잠금 (4.5) |
| `votes` | SELECT (주최자): 자기 방 전체 / (참여자): 자신이 쓴 표만 | 익명성 보장 (4.4) |
| `messages` | INSERT: sender_id = auth.uid() | 본인만 작성 |
| `messages` | SELECT: 송수신자 + 주최자 | 익명성 + 주최자 열람 권한 |

> RLS 정책은 SQL로 선언, 변경 시 마이그레이션. 코드 검수 시 정책 누락 = 보안 사고로 간주.

## 5. 기능 요약

### 5.1 주최자 기능
- **회원가입 / 로그인 (Supabase Auth, 이메일+비밀번호 + Google OAuth 옵션) ⭐ v6**

  - [다음 단계로] (5초 카운트 모달 v4)
  - **[이전 단계로] (재오픈/읽기전용 모드 선택 v4)**
  - [⚙ 단계 구성 편집] (잠금 모드, 현재 단계 삭제 가능 v4)

- 신고 처리
- 결과 통계 + CSV 내보내기

### 5.2 참여자 기능
- **QR 진입 → 약관 동의 → 닉네임 등록 → Anonymous Auth 자동 로그인 ⭐ v6**
  - 회원가입 / 비밀번호 / 이메일 입력 일체 없음
- 약관에 **"주최자 데이터 열람" 명시 (v4)**
- 약관에 **"한 기기에서 계속 사용해 주세요" 안내 (v6 신규, 세션 한계 고지)**
- 단계별 진행:
  - 호감도 단계: 참여자 목록 → ❤ (한도 내) 또는 "투표 안 함"
  - 호감도 단계 진행 중 신규 입장 차단 (v4)
  - 비호감도 단계: "🙂 자유 시간"
- 자기 자신 선택 불가
- ❌ 차단 기능 제거 (v4) — 신고만 유지
- 잠금 화면 (주최자 편집 중)
- 매칭 성공 알림 / 쪽지 / 노래 요청

## 6. 시스템 구성 (v6 변경 — Supabase 기반)

| 레이어 | 스택 | 변경 |
|--------|------|------|
| 프론트엔드 | Next.js 14 (App Router) + Tailwind | v3 유지 |
| 인증 | **Supabase Auth (주최자: 이메일+OAuth, 참여자: Anonymous)** | ⭐ v6 변경 (자체 JWT → Supabase) |
| DB | **Supabase Postgres** (managed) | ⭐ v6 변경 (자체 PG → Supabase) |
| 권한 | **Postgres RLS (Row Level Security)** | ⭐ v6 신규 (백엔드 코드 대체) |
| 실시간 | **Supabase Realtime** (SSE 대체, Postgres CDC 기반) | ⭐ v6 변경 |
| 서버 로직 | Supabase **Edge Functions** (Deno, 매칭 판정/CSV 익스포트만) | ⭐ v6 변경 (NestJS 서버 제거) |
| 스토리지 | Supabase Storage (CSV 임시 저장) | ⭐ v6 신규 |
| 배포 | Vercel (프론트) + Supabase Cloud (백엔드) | ⭐ v6 변경 (Railway 제거) |

**서버 로직 최소화 원칙**: CRUD/권한은 RLS로, 실시간은 Realtime으로 처리. Edge Function은 다음 3가지에만 사용:
1. 매칭 판정 (마지막 호감도 단계 종료 시 트랜잭션)
2. CSV 익스포트 (전체 데이터 집계)
3. QR 코드 발급 (서명된 room URL)

## 7. 핵심 데이터 모델 (v6 변경 — Supabase 기반)

```sql
-- v6 신규: Supabase Auth 내장 테이블 활용
auth.users (Supabase 기본)         -- 주최자 + 참여자 모두 여기에 저장
  - id (UUID)                      -- 주최자: 이메일 가입, 참여자: anonymous
  - email (주최자만), is_anonymous: bool

profiles (v6 신규)                  -- 주최자 추가 정보
  - id (FK auth.users)
  - display_name, avatar_url
  - created_at
  -- RLS: 본인만 read/update

Room
  - id, host_id (FK auth.users), name, max_participants  -- v6: host_id 명시 FK
  - status: 'pending'|'live'|'editing'|'closed'
  - is_locked_for_editing: bool

Stage
  - id, room_id, order, name, description?
  - collect_vote: bool
  - max_selections: int                    -- v4: 호감도 단계 다중선택 한도 (1~10)
  - trigger_timer_minutes: int?
  - trigger_all_voted: bool
  - trigger_manual: bool
  - status: 'pending'|'active'|'closed'|'reopened'|'view-only'|'deleted'
  - deleted_at: timestamp?                 -- v5 신규: 삭제 시각 (영구 기록)
  - current_run_id: int                    -- v5 신규: 현재 실행 회차 (재오픈마다 +1)

StageRun (v5 신규 테이블)               -- 재오픈 추적용
  - id (= stage_run_id)
  - stage_id (FK)
  - run_number: int (1, 2, 3... 재오픈마다 증가)
  - opened_at, closed_at
  - mode: 'normal'|'reopened'|'view-only'
  - allow_join: bool                       -- v5: 정책 일관 적용
                                              · normal + collect_vote=true → false
                                              · reopened + collect_vote=true → false (v5 정합성)
                                              · view-only → 입장 무관 (투표 자체 불가)
                                              · normal + collect_vote=false → true
  - is_final_for_matching: bool            -- v5: 매칭 판정에 사용되는 run인지
                                              · 마지막 호감도 단계의 가장 최근 run만 true

Participant
  - id, room_id, nickname, entry_number
  - anon_user_id (FK auth.users)           -- v6 변경: session_token → Supabase Anonymous Auth JWT의 sub
  - status: 'active'|'kicked'|'left'       -- v4: 신규
  - kicked_at: timestamp?
  -- UNIQUE (room_id, anon_user_id)        -- v6: 한 사람이 같은 방에 중복 입장 방지

Vote (영구 잠금, 다른 참여자에게 익명)
  - id
  - stage_run_id (FK StageRun)             -- v5 변경: stage_id → stage_run_id
  - voter_id, target_id (nullable = "투표 안 함")
  - is_locked: bool (run 종료 시 true)
  - created_at

  ※ 매칭 계산: StageRun.is_final_for_matching=true인 run의 Vote만 사용
  ※ 재오픈 1회 → 새 StageRun 생성 (run_number=2)
     - 이전 run의 Vote는 보존 (분석/통계용)
     - 새 run에서 다시 투표 가능 (참여자 입장에서는 "재투표")
  ※ 재오픈 N회 → 가장 최근 run이 매칭 기준

Match
  - id, room_id, participant_a_id, participant_b_id
  - matched_stage_id (마지막 호감도 단계)
```
